## 一、协议适用范围

本协议适用于 `Kite Desktop` 桌面应用中“应用数据分析”相关功能的隐私说明。

本协议仅说明应用数据分析功能可能涉及的信息处理行为，不当然覆盖以下内容：

1. 用户自行管理的 Kubernetes 集群业务数据
2. 用户自行输入的文本、日志、配置文件内容
3. 应用更新检查、下载更新等其他联网能力
4. 第三方服务本身的隐私政策

如果产品中的其他能力涉及独立的信息处理规则，应由相应功能另行说明。

---

## 二、我们为什么使用应用数据分析

我们使用应用数据分析功能，主要是为了持续改进 `Kite Desktop` 的产品体验，包括但不限于：

1. 了解哪些模块和功能被更频繁地使用
2. 评估 AI 对话、终端、kubeconfig 导入、应用内更新等桌面专属能力的实际价值
3. 发现用户流程中的高频路径、低使用模块或明显的交互摩擦
4. 为后续产品优化、界面调整和功能规划提供聚合层面的依据

我们使用应用数据分析的目的，不是为了识别用户管理的具体业务对象，也不是为了收集用户输入内容或敏感资源信息。

---

## 三、我们可能收集的信息类型

在应用数据分析功能开启的前提下，我们可能收集以下类型的脱敏信息：

### 1. 页面类型信息

我们可能记录用户访问了哪一类功能页面，但不会记录真实资源对象路径。

例如：

- `overview`
- `settings`
- `pods/list`
- `pods/detail`
- `deployments/detail`
- `ai-chat`

这类信息用于分析模块使用热度与功能入口分布。

### 2. 功能事件信息

我们可能记录少量关键功能动作，例如：

1. 切换集群
2. 打开 AI 对话
3. 打开 kubectl 终端
4. 导入 kubeconfig
5. 点击检查更新
6. 开始下载更新
7. 开始安装更新

为降低隐私风险，这类事件的附带信息将尽量限制为固定枚举值、布尔值或分类值，例如：

- `entry=button`
- `runtime=desktop`
- `mode=file_dialog`

---

## 四、我们不会收集的信息

为了尽量降低隐私风险，我们的应用数据分析不应收集以下信息：

1. 集群名称
2. 命名空间名称
3. Kubernetes 资源名称
4. AI 会话 ID
5. 用户输入的 Prompt、对话内容或编辑内容
6. Pod 日志、终端输出、YAML 配置内容
7. 本地文件路径、本地目录路径
8. 下载地址、外部 URL 的完整值
9. 密码、Token、证书、API Key 等凭据
10. 任何可直接识别具体业务对象或用户输入内容的高敏感信息

---

## 五、页面 URL 的处理方式

`Kite Desktop` 属于桌面应用，很多页面在路由层可能包含 Kubernetes 对象上下文。

为了避免对象级信息进入分析系统，我们会尽量遵循以下原则：

1. 不直接上传真实页面 URL
2. 不上传 query string
3. 不上传 hash
4. 页面访问数据会优先被归一化为脱敏后的页面类型标识，再进入分析系统

例如：

- 真实页面：`/pods/default/nginx-7d8f9`
- 脱敏后页面类型：`pods/detail`

---

## 六、数据分析功能的基础设施

当前版本的 `Kite Desktop` 计划使用自建 Umami 服务承载应用数据分析。

在该方案下，应用数据分析相关请求会发送至产品维护者自建的数据分析服务，而非第三方官方托管云服务。

如未来分析基础设施发生变化，我们将根据实际情况更新本协议内容。

---

## 七、数据分析功能的开启与关闭

`Kite Desktop` 中的应用数据分析属于可控功能。

原则上：

1. 用户可以在应用设置中启用或关闭“数据分析”
2. 当用户关闭该功能后，应用不应继续发送相关应用数据分析请求
3. 若用户重新启用该功能，应用数据分析能力可重新生效

若产品实现需要在切换开关后刷新页面或重启相关模块才能完全生效，我们将尽量在界面中做出清晰说明。

---

## 八、Do Not Track 与最小化原则

在应用数据分析能力设计中，我们将尽量遵守以下原则：

1. 尊重 `Do Not Track` 等用户隐私偏好
2. 默认不采集与产品改进无直接关系的信息
3. 仅保留少量必要、低敏感的页面类型与关键事件
4. 优先采用脱敏、归一化后的统计标识
5. 尽量减少分析事件数量，避免无边界扩张

---

## 九、应用数据分析与版本检查的区别

应用数据分析与版本检查属于两类不同的联网行为。

### 1. 应用数据分析

用于理解产品功能使用情况与改进方向。

### 2. 版本检查

用于检查应用是否存在可用新版本，可能访问 GitHub Releases 或其他更新源。

因此：

1. 关闭“数据分析”不必然关闭“版本检查”
2. 关闭“版本检查”也不必然关闭“数据分析”
3. 两者应在设置项与产品说明中分别展示

---

## 十、数据用途限制

通过应用数据分析获得的信息，仅应用于以下目的：

1. 分析功能使用趋势
2. 评估产品模块价值
3. 发现功能入口与流程设计问题
4. 辅助产品优化和迭代决策

我们不应将应用数据分析数据用于：

1. 识别用户管理的具体业务系统结构
2. 还原具体对象内容
3. 分析用户输入内容
4. 构建与业务对象强绑定的精细画像
