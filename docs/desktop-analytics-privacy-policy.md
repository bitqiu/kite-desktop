## 1. 我们为什么需要应用数据分析

`Kite Desktop` 是一款桌面端 Kubernetes 管理工具。

我们引入应用数据分析的目的，是为了理解以下问题：

1. 用户更常使用哪些功能模块。
2. 哪些桌面专属能力更有使用价值，例如 AI 对话、终端、kubeconfig 导入、应用内更新等。
3. 哪些产品流程存在明显摩擦或低使用率，从而帮助后续改进体验。

应用数据分析的目的**不是**采集用户管理的 Kubernetes 业务对象内容，也**不是**追踪用户输入、操作细节或敏感业务数据。

---

## 2. 数据分析功能的开关与控制

在 `Kite Desktop` 中，应用数据分析是一个可由用户控制的独立功能。

原则上：

1. 用户可以在设置中启用或关闭“数据分析”。
2. 用户关闭该功能后，应用不应继续向分析系统发送相关统计数据。
3. “数据分析”和“版本检查”属于两类不同的联网行为：
   - `数据分析`：用于产品使用统计
   - `版本检查`：用于检查是否存在新版本更新

关闭数据分析不会自动关闭版本检查；关闭版本检查也不会自动关闭数据分析。

---

## 3. 我们会收集哪些信息

按照当前拟定方案，应用数据分析仅收集以下低敏感、脱敏后的信息：

### 3.1 页面类型信息

我们会记录用户进入了哪一类页面，但不会记录真实对象标识。

示例：

- `overview`
- `settings`
- `pods/list`
- `pods/detail`
- `deployments/detail`
- `ai-chat`

这类信息用于理解模块使用热度。

### 3.2 功能事件信息

我们可能会记录少量关键功能动作，例如：

- 切换集群
- 打开 AI 对话
- 打开 kubectl 终端
- 导入 kubeconfig
- 点击检查更新
- 开始下载更新
- 开始安装更新

这类事件的属性会尽量限制为枚举值或布尔值，例如：

- `entry=button`
- `runtime=desktop`
- `mode=file_dialog`

---

## 4. 我们不会收集哪些信息

为了降低隐私风险，`Kite Desktop` 的应用数据分析不应收集以下内容：

1. 集群名称
2. 命名空间名称
3. Kubernetes 资源名称
4. AI 会话 ID
5. 用户输入的 Prompt 或对话内容
6. Pod 日志、YAML 内容、终端输出内容
7. 本地文件路径、本地目录路径
8. 下载链接、外部 URL 的完整值
9. 用户账号密码、API Key、Token、证书等凭据
10. Kubernetes 对象详情、业务数据、告警文本等高敏感信息

换句话说，应用数据分析只用于理解产品使用模式，不用于读取或上送用户管理的业务资源内容。

---

## 5. URL 与页面访问数据的处理方式

`Kite Desktop` 属于桌面应用，很多页面路由会包含 Kubernetes 资源上下文。

为了避免在统计中暴露对象级信息，应用数据分析采用以下原则：

1. 不直接上传真实页面 URL。
2. 不上传 query string。
3. 不上传 hash。
4. 页面访问会先被归一化成脱敏 page key，再进入分析系统。

例如：

- 真实页面：`/pods/default/nginx-7d8f9`
- 归一化后：`pods/detail`

---

## 6. 数据分析基础设施

`Kite Desktop` 当前计划使用自建 Umami 实例承载应用数据分析。

当前目标分析脚本来源：

- `https://umami.eryajf.net/script.js`

这意味着分析数据会发送至应用维护者自建的分析服务，而不是第三方托管的官方云服务。

---

## 7. Do Not Track 与最小化原则

在拟定方案中，我们会尽量遵守以下隐私约束：

1. 尊重浏览器或运行环境中的 `Do Not Track` 偏好。
2. 禁止自动采集真实 URL。
3. 仅保留少量高价值、低敏感事件。
4. 使用脱敏后的页面类型替代真实资源路径。
5. 不采集任何业务资源内容。

我们遵循“数据最小化”原则：只保留为改善产品体验所需的最低限度信息。

---

## 8. 数据用途

收集到的应用数据分析信息仅用于以下目的：

1. 了解产品功能使用趋势
2. 判断桌面专属能力是否被使用
3. 发现功能入口设计是否合理
4. 辅助规划产品优化优先级
5. 评估应用内更新、AI 对话、终端等能力的实际价值

这些数据不应用于：

1. 推断用户管理的业务系统内部结构
2. 恢复具体资源对象内容
3. 分析用户输入文本或业务日志
4. 与用户身份做精细画像

---

## 9. 与版本检查的区别

`Kite Desktop` 可能会提供版本检查功能，用于访问 GitHub Releases 或其他更新源以确认是否存在新版本。

这类行为属于“软件更新能力”，不属于应用数据分析。

因此：

1. 数据分析开关控制的是产品使用统计。
2. 版本检查开关控制的是是否自动或手动联网检查更新。
3. 两者应在设置项与产品说明中分别展示。

---

## 10. 数据保留与后续调整

本文件为草案，重点是先明确产品边界与隐私原则。

后续如果以下任一事实发生变化，应同步更新本文：

1. 分析系统发生变化
2. 采集事件范围发生变化
3. 默认开关策略发生变化
4. 页面统计方式发生变化
5. 版本检查与数据分析的边界发生变化

如果未来产品加入更多数据分析能力，也应继续遵循本文中的脱敏与最小化原则。

## 12. 建议在正式发布前再次确认的事项

在将本文作为正式对外文档前，建议再次确认：

1. 实际代码实现是否已经采用脱敏 page key，而不是自动真实 URL。
2. 实际 analytics 开关关闭后，是否确实完全停止发送 Umami 请求。
3. 是否已经恢复对 query/hash/DNT 的保护配置。
4. 是否已经明确 analytics 与版本检查为独立设置项。
5. 默认值是否与用户看到的说明一致。

如果这些事项尚未落地，本文应继续标记为“草案”。
